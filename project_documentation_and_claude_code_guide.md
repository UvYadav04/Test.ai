# PART A — PROJECT DOCUMENTATION

## 1. What This Project Is

A workspace-based platform where users:

1. Create a **workspace**.
2. Upload **private organizational data** — initially CSV, JSON, and PDF files.
3. Ask **natural language questions** or request **reports** about that data.
4. Get back answers/reports generated by a system of specialized AI agents that investigate the data, backed by deterministic tools (not hallucinated), with full traceability back to source files.

The core promise: **no answer is given without evidence traced back to an actual file the user uploaded.**

## 2. Why This Architecture (Design Philosophy)

### 2.1 Core Design Principle

> Functions perform deterministic operations. LLM calls perform bounded semantic reasoning. Agents are introduced only when a task requires an independent, adaptive observe-act-observe loop.

This principle governs every decision in the system. Concretely:

| Type | Examples | Why |
|---|---|---|
| **Deterministic function/tool** | DuckDB query execution, Parquet read/write, chunking, vector upsert | No judgment needed — same input always produces same output |
| **Bounded LLM call** | Hypothesis generation, (future) synthesis step | One-shot reasoning, no tool use, no loop |
| **Agent** | Main Orchestrator, Tabular Agent, Document Agent | Needs to observe results, decide next action, retry, and adapt — a real loop |

We do **not** turn every LLM-touching step into an "agent." Over-agentifying a system creates unnecessary cost, latency, and unpredictability. Agents are reserved for genuinely adaptive investigation loops.

### 2.2 Context Isolation

Every agent runs in its own **isolated context window**. This means:

- The Main Orchestrator never sees a subagent's raw tool call trace, raw query results, or raw retrieved document chunks.
- Subagents never see each other's work, and never see the Main Orchestrator's full conversation history.
- Subagents return only a **compact, structured result** (summary, findings, limitations, confidence, references) — never raw data dumps.

Why: this keeps the Main Orchestrator's context clean and scalable. A user could ask a question that requires the Tabular Agent to run 15 exploratory queries — none of that noise should pollute the orchestrator's reasoning about the overall investigation.

### 2.3 Why Private Data Requires This

Because uploaded data is **private and organizational**, every design decision defaults to conservative, traceable behavior:

- Agents only ever see files explicitly assigned to them by the orchestrator — never the whole workspace.
- Every finding an agent returns must carry an `artifact_ref`/`source_ref` back to a real file/row/chunk — no unattributed claims.
- Raw data never gets summarized away without a way to trace back to its origin (this is why the Artifact Store exists — see §6).

## 4. Component Reference

### 4.1 Main Orchestrator

**Responsibilities:**
- Understand the user's objective and constraints.
- List and select relevant workspace files (via a lightweight file catalog — see §4.4).
- Decide whether tabular analysis, document investigation, or both are required.
- Decide whether hypothesis generation is useful (for complex "why" queries).
- Delegate investigations to specialized agents.
- Receive only compact structured findings back.
- Track the investigation in a structured **Investigation State** object.
- Decide whether more evidence is required, or whether to finalize.
- Generate the final answer, chart, or report.

**Does NOT:**
- Run raw SQL/DuckDB queries itself.
- Run RAG searches itself.
- See raw tool traces or chunk content from subagents.

### 4.2 Tabular Agent

Handles CSV (and later JSON, XLSX, SQLite) data analysis.

**Tools available to this agent:**
```
list_allowed_files()
inspect_schema()
sample_rows()
find_join_candidates()
query_data()
aggregate()
validate_result()
```

**Behavior:** Runs its own internal loop — inspect schema, sample data, run queries, validate results, retry if needed, perform cross-file joins — entirely within its own isolated context, before returning one final structured result.

**Invocation contract:**
```
invoke_tabular_agent(
    objective: str,
    assigned_files: list[FileRef],
    constraints: dict,
) -> TabularFindings
```

**Return shape:**
```
TabularFindings
  summary: str
  findings: [ { statement, columns_used, computation, artifact_ref } ]
  limitations: str
  confidence: "high" | "medium" | "low"
  artifact_refs: [ref]
```

### 4.3 Document Agent

Handles PDF (and later other document types) investigation.

**Tools available to this agent:**
```
search_documents()
get_chunk()
get_surrounding_chunks()
search_within_file()
```

**Behavior:** Performs multiple RAG searches, expands queries, follows discovered concepts across documents, searches for contradictions, and returns compact evidence-backed findings. Raw RAG chunks stay inside this agent's context — never passed up to the orchestrator.

**Invocation contract:**
```
invoke_document_agent(
    objective: str,
    assigned_files: list[FileRef],
    constraints: dict,
) -> DocumentFindings
```

**Return shape:**
```
DocumentFindings
  summary: str
  findings: [ { statement, source_file, chunk_refs, artifact_ref } ]
  limitations: str
  confidence: "high" | "medium" | "low"
  artifact_refs: [ref]
  source_refs: [ { file, page_or_section } ]
```

### 4.4 File Catalog (Orchestrator-Level, Shallow)

Before delegating, the orchestrator needs a **shallow** view of workspace files — filenames, file type, and a lightweight schema summary (columns for tabular, page count for PDFs) — without loading full file content. This is distinct from an agent's own deep `inspect_schema()`/`extract_metadata()` calls, which happen only after a file is assigned to that agent.

This shallow catalog is populated at ingestion time (see §5) and stored as workspace metadata.

### 4.5 Hypothesis Generation

An **optional, bounded LLM call** (not an agent) that the orchestrator can invoke for complex or "why"-style queries:

```
Simple Query        → Direct Investigation
Complex/Why Query   → Generate Hypotheses → Prioritize → Delegate to Agents
```

No tool use, no loop — a single reasoning call that helps the orchestrator prioritize which investigation directions matter most before delegating.

### 4.6 Investigation State

Every user request creates one Investigation State object — the **canonical structured working memory** of the orchestrator for that request.

```
InvestigationState
  session_id
  objective
  constraints
  selected_files
  active_tasks
  completed_tasks
  findings
  open_questions
  status
```

**Important behavior (per project decisions):** Both the Tabular Agent and Document Agent run and prepare their full results independently. **Neither writes to Investigation State until both have returned.** Once both are back, their outputs are both written into Investigation State together. This avoids partial/interleaved writes and keeps state updates atomic per investigation round.

### 4.7 Agent Context Isolation & Artifact Store

Subagents never leak raw internals into the orchestrator's context. Detailed outputs (full query results, full retrieved chunks, computations, charts, reports) are persisted to an **Artifact Store**, separate from any agent's context. Agents and the orchestrator work with compact summaries and `artifact_ref`s; the full artifact is only loaded when explicitly required (e.g., final report rendering, or user drill-down).

### 4.8 Main Agent Memory Management

The orchestrator's context is composed of:
```
System Prompt
User Objective + Constraints
Investigation State
Cumulative Memory Summary
Recent Unsummarized Messages
```

When context utilization crosses roughly 80%, a **Memory Pruner** runs: it takes the previous summary + older messages + current Investigation State, and produces an updated cumulative summary, after which older messages are dropped. **Investigation State itself is never replaced by the summary** — it remains structured and durable. (Note: because both agents return compact summaries rather than raw data, this pipeline fills context slowly in practice — pruning is a safety net, not something expected to trigger frequently.)

### 4.9 Final Runtime Flow

```
User Query
    → Create Investigation State
    → Main Orchestrator
    → List/Filter Workspace Files
    → Choose Investigation Strategy
         ├── Tabular Agent → DuckDB Tools → Structured Findings
         ├── Document Agent → RAG Service → Structured Findings
         └── Optional Hypothesis Generation → Additional Investigations
    → Update Investigation State (once both agents have returned)
    → Enough Evidence?
         ├── No → Continue Investigation
         └── Yes → Generate Answer / Chart / Report
```

## 5. Data Ingestion Pipeline

Before any agent can work, uploaded files must be normalized:

- **CSV / JSON** → converted to **Parquet** (queryable via DuckDB later).
- **PDF** → chunked and stored in a **Vector DB** (Chroma now, Pinecone later) for RAG retrieval.

### 5.1 Directory Structure

```
project_root/
├── ingestion/
│   ├── manager.py                # IngestionManager
│   ├── registry.py               # extension -> ingestor class mapping
│   ├── file_types/
│   │   ├── base.py                # BaseIngestor abstract class
│   │   ├── csv/
│   │   │   ├── csv_ingestor.py
│   │   │   └── utils.py
│   │   ├── json/
│   │   │   ├── json_ingestor.py
│   │   │   └── utils.py
│   │   └── pdf/
│   │       ├── pdf_ingestor.py
│   │       ├── chunker.py
│   │       └── utils.py
│   └── storage/
│       ├── base.py               # BaseObjectStore abstract class
│       └── local_store.py        # LocalParquetStore
│
├── vectordb/                     # SEPARATE from ingestion — it's also queried directly by agents later
│   ├── base.py                   # BaseVectorStore abstract class
│   ├── schema.py                 # ChunkRecord
│   └── chroma_store.py           # ChromaVectorStore
│
└── tests/
    ├── test_csv_ingestor.py
    ├── test_json_ingestor.py
    ├── test_pdf_ingestor.py
    └── test_manager.py
```

**Why VectorDB is separate from Ingestion:** ingestion only *writes* to the vector DB. But later, the Document Agent will *read* from it directly (via `search_documents()` etc.). If VectorDB were nested inside `ingestion/`, that read path would awkwardly import "up" through an ingestion-labeled module. Keeping it as a top-level sibling reflects that it's a shared service, not an ingestion-only artifact.

### 5.2 Registry Pattern

`registry.py` maps file extensions to ingestor classes so `IngestionManager` never branches on file type directly:
```python
REGISTRY = {
    ".csv": CSVIngestor,
    ".json": JSONIngestor,
    ".pdf": PDFIngestor,
}
```
Adding a new file type later (e.g. `.xlsx`) means adding a new folder + one registry line — never touching the manager.

### 5.3 Core Schemas

```python
@dataclass
class IngestionResult:
    file_id: str
    workspace_id: str
    status: Literal["success", "partial", "failed"]
    output_ref: str            # parquet path OR vectordb collection ref
    schema_summary: dict       # tabular: {columns, dtypes}; pdf: {page_count, sections}
    row_count: Optional[int] = None
    chunk_count: Optional[int] = None
    errors: list = field(default_factory=list)

@dataclass
class ChunkRecord:
    chunk_id: str
    file_id: str
    workspace_id: str
    text: str
    embedding: Optional[list] = None
    metadata: dict = None      # {page, section, chunk_index, source_file}
```

## 6. Homescreen "Live Architecture" Visualization

To build user trust, the homescreen/query UI shows a **live activity panel** reflecting what's actually happening during an investigation — not a static marketing diagram, but a real-time view driven by each agent's compact `summary`/status updates.

**Principles:**
1. Show plain-language status ("Querying churn.csv for Q3 records"), not raw tool logs.
2. Show which files each agent is touching — a trust signal for private data.
3. Collapse to one line by default; expandable for detail.
4. Persist as a collapsible "how I got this answer" trail attached to the final report — this doubles as an audit/explainability feature for enterprise buyers.
5. Color-code by agent type (e.g., blue = deterministic/Tabular, purple = semantic/Document) to visually reinforce the deterministic-vs-agentic design principle.

This is a v1.x UI feature layered on top of the backend — it consumes the same `summary`/`status` fields agents already produce, so no new backend contract is needed beyond streaming those updates to the frontend as they occur.

## 7. Explicitly Deferred to Later Versions (Not in Scope for Initial Build)

- **Synthesis / conflict-resolution step**: a bounded LLM call that runs after both agents return, to explicitly label relationships between findings (`supports`, `contradicts`, `partial_overlap`, `unrelated`, `temporal_mismatch`) before final answer generation, and to gate the "enough evidence?" decision on unresolved conflicts. Deferred until the foundation is built and working end-to-end.
- **Document Agent evidence verification**: a faithfulness check confirming claimed quotes/statements actually appear in the referenced chunk (mirrors `validate_result()` on the tabular side). Planned, not in the initial build.
- **JSON/XLSX/SQLite tabular support beyond CSV**, **Pinecone backend**, **object storage (S3) backend** — interfaces are designed to support these via swap-in implementations, but initial build targets local Parquet + Chroma + CSV/JSON only.
